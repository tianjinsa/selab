# 校园智能生活服务平台需求规格说明书

版本：V1.0
日期：2026-06-30

本文是校园智能生活服务平台的当前需求规格与实现状态说明，面向软件工程课程设计验收。

## 状态标记

| 标记 | 含义 |
| --- | --- |
| `[DONE]` | 已有前后端或管理端实现，可用于课程演示。 |
| `[PARTIAL]` | 主流程已实现，但覆盖范围、页面入口、真实集成或边界流程不完整。 |
| `[MOCK]` | 按课程演示方式模拟实现，不等同于真实生产能力。 |
| `[PENDING]` | 当前代码未发现可用实现。 |
| `[UNVERIFIED]` | 需求合理，但尚未通过压测、安全测试或生产环境验证。 |

## 仓库结构

| 目录 | 说明 | 状态 |
| --- | --- | --- |
| `weixin/` | 微信小程序端，包含任务互助、二手市场、社区论坛、智能体、我的、消息与聊天等页面。 | `[DONE]` |
| `web/` | 管理员后台，Vue 3 + Element Plus，构建产物输出到 `server/public`。 | `[DONE]` |
| `server/` | Node.js + Express 后端，提供统一认证、任务、社区、二手、智能体、消息、举报和后台管理 API。 | `[DONE]` |
| `server/sql/` | SQL Server 建表脚本。当前运行默认使用 JSON 演示数据仓库。 | `[PARTIAL]` |

技术栈：

- 小程序端：微信小程序 + TDesign MiniProgram。
- 管理后台：Vue 3 + Element Plus + Vite。
- 后端：Node.js + Express + JWT + WebSocket (`ws`)。
- 数据：默认 `server/data/db.json` 演示仓库；提供 SQL Server 初始化脚本。

## 1. 项目总述

### 1.1 背景

平台面向校园生活服务场景，整合任务互助、社区交流、二手交易、智能咨询与统一消息能力，解决校园内信息分散、互助效率低、闲置资源流转难等问题。

### 1.2 目标

建设“校园智能生活服务系统”统一底座，提供统一用户体系、统一服务入口、统一消息中心与统一后台管理能力，并支撑任务互助、社区论坛、二手市场和校园信息智能体等子系统。

### 1.3 开发阶段

| 阶段 | 子系统 | 当前状态 |
| --- | --- | --- |
| P1 | 主系统底座 | `[DONE]` |
| P2 | 校园任务互助系统 | `[PARTIAL]` |
| P3 | 校园社区论坛系统 | `[PARTIAL]` |
| P4 | 校园二手市场系统 | `[PARTIAL]` |
| P5 | 校园信息智能体系统 | `[PARTIAL]` |

证据入口：

- 后端路由：`server/src/routes/*.js`
- 小程序页面：`weixin/pages/**`
- 管理后台：`web/src/App.vue`
- 数据结构与演示数据：`server/src/seed.js`
- SQL Server Schema：`server/sql/schema.sql`

## 2. 校园智能生活服务系统

### 2.1 定位

主系统是统一入口和公共底座，承担用户身份、账号资料、消息通知、私信沟通、子系统导航和个人数据汇聚职责。

### 2.2 功能需求

| 需求 | 状态 | 当前实现 |
| --- | --- | --- |
| 学号/手机号注册、密码登录 | `[DONE]` | 后端提供 `/api/auth/register`、`/api/auth/login`；小程序登录页和邀请注册页已接入，注册页校验邀请码、账号、昵称、密码、手机号和学号。 |
| 微信快捷登录 | `[PARTIAL]` | 后端提供 `/api/auth/wechat-login` 模拟 openId 注册登录；未接入真实 `wx.login` code 换取 openId。 |

| 校园身份校验 | `[MOCK]` | 用户字段 `schoolVerified: true`，当前使用模拟校验。 |
| 全平台单点登录 | `[DONE]` | JWT 在所有 API 中复用，管理后台和小程序均使用 Bearer Token。 |
| 个人资料管理 | `[DONE]` | `/api/auth/me`、`/api/users/me` 与小程序资料编辑页已实现。 |
| 密码修改 | `[PARTIAL]` | 后端提供 `/api/auth/password`，小程序设置页偏展示，未形成完整修改密码表单。 |
| 异常登录检测与风险提醒 | `[PENDING]` | 未发现登录风险检测、设备指纹或异常提醒逻辑。 |

| 架构扩展能力 | `[UNVERIFIED]` | 课程验收阶段以核心流程演示为目标；生产级万级并发需切换真实数据库、增加连接池/缓存/压测后再确认。 |
| 密码加密存储 | `[DONE]` | 使用 `bcryptjs` 哈希。 |
| 敏感信息脱敏展示 | `[PARTIAL]` | 默认种子数据和注册逻辑对手机号做基础脱敏。 |

| 任务浏览、筛选、关键词搜索 | `[PARTIAL]` | 后端支持类型、关键词、状态、酬金区间；小程序支持类型和关键词，并提供加载态与空态。 |

## 6. 校园信息智能体系统

### 6.1 定位

课程验收阶段定位为基于 OpenAI 兼容 Chat Completions、本地知识库和只读工具调用的校园问答助手，可演示模型问答、来源展示、流式会话历史和跨子系统检索。向量语义检索不纳入当前验收范围。

### 6.2 功能需求

| 需求 | 状态 | 当前实现 |
| --- | --- | --- |
| 知识库构建与管理 | `[PARTIAL]` | 后端支持创建/更新知识条目，后台展示列表；删除和完整编辑 UI 不完整。 |
| 本地知识库检索 | `[DONE]` | 当前使用关键词计分匹配知识库条目，并写入模型系统上下文。 |
| 向量语义检索 | `[PENDING]` | 未接入 embedding、向量数据库或真实语义召回，不纳入当前验收范围。 |
| 调用任务、二手、社区能力 | `[DONE]` | 模型可调用 `search_tasks`、`search_goods`、`search_posts`、`get_platform_taxonomy` 只读工具，后端仍保留 `/api/agent/functions/:tool` 查询入口。 |
| 工具调度能力 | `[DONE]` | 后端向模型传入 OpenAI 兼容 `tools` 定义，并把工具结果回传给模型生成最终回答。 |
| 多场景问答 | `[DONE]` | 小程序提供校园/任务/交易助手切换，后端根据 `agentKey` 注入不同场景提示词。 |
| 多轮对话 | `[DONE]` | `/api/agent/runs` 会保存会话消息，并把最近历史消息带入模型上下文。 |
| 答案溯源 | `[DONE]` | 回复包含知识库来源。 |
| 会话历史留存 | `[DONE]` | `/api/agent/sessions` 已实现。 |
| 数据统计 | `[DONE]` | 后台统计问答量、会话数、知识库数、提示词数。 |
| 场景化提示词配置 | `[DONE]` | 后端支持创建提示词，种子数据含多场景提示词，active prompt 会写入模型系统上下文。 |

### 6.3 Agent 模式能力

### 6.4 非功能需求

智能体响应时间、准确率、回答安全边界和隐私合规均作为待测指标。当前实现适合课程演示，不承诺生产级大模型准确率；使用外部模型前需要配置 `OPENAI_BASE_URL`、`OPENAI_API_KEY` 和 `OPENAI_MODEL`，并补充评测集、安全策略和日志审计。

## 8. 演示边界

- 当前默认数据仓库是 `server/data/db.json`。SQL Server 建表和初始化脚本已提供，但业务读写服务尚未切换到 SQL Server。
- 支付、资金托管、退款和到账均为模拟实现。
- 微信登录、校园身份校验和图片上传均未接入真实外部平台闭环。
- 所有性能指标和并发目标均未通过自动化压测验证。
- 管理后台已覆盖审核和参数配置主流程，并提供响应式控制台、加载态和空态；部分知识库、提示词、订单和用户操作仍以列表或简化按钮为主。

## 9. 需求合理性调研摘要

已使用 `smart-search-cli` 和 `find-docs`/Context7 查验技术与平台能力。当前结论：

- Express 4、`ws`、Vue 3、Element Plus、Vite、TDesign MiniProgram 用于课程设计项目是合理组合。
- 微信小程序正式环境网络请求要求 HTTPS/WSS 域名配置、证书和域名规则；`localhost` 只适合开发者工具或本地演示。
- 微信支付 `wx.requestPayment` 需要小程序平台开通微信支付并由后端提供预支付参数；当前模拟支付应明确标注为课程演示。
- “万级并发”“准确率 >= 90%”“强一致资金结算”等生产级指标不纳入当前验收范围，当前列为待测能力边界。
